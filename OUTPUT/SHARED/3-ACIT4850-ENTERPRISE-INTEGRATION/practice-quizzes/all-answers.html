Multi-SelectQ1 / 30C, D, F
1. Select THREE characteristics of on-premise infrastructure.
Explanation: On-premise = you own the hardware. You Pay Upfront (Capital Expense) and it's Controlled By Your Organization. Cloud = Operational Expense (monthly billing), Pay As You Go, Controlled by a 3rd Party.
Multi-SelectQ2 / 30B, E, F
2. Select THREE characteristics of cloud infrastructure.
Explanation: Cloud = 3rd party controls hardware, you pay monthly (Operational Expense), and you Pay As You Go. On-premise = Capital Expense + Pay Upfront + You control it.
MCQ3 / 30C
3. What capability does Confluence provide?
Explanation: Confluence = Knowledge Base (documentation, wikis, meeting notes). JIRA = Work Management (tasks, backlogs, bug tracking). These are both Atlassian products often deployed together.
Multi-SelectQ4 / 30B, D, E, F
4. Which installation methods are valid for a PRODUCTION installation of JIRA or Confluence? Select all that apply.
Explanation: Production install methods: Installer-Linux, Installer-Windows, Cluster (HA), Zip/Archive. Docker Container = NOT for production. Installer-Mac = not available. This was a quiz question Nick got wrong — exam trap!
Multi-SelectQ5 / 30A, C, D, E
5. What database options are valid for a PRODUCTION installation of JIRA or Confluence? Select all that apply.
Explanation: Supported production DBs: Oracle, PostgreSQL, MySQL, Microsoft SQL Server. Embedded H2 = development/evaluation database ONLY — never use in production.
MCQ6 / 30B
6. Select the best definition of a Functional Requirement.
Explanation: Functional Requirement = a function of a system/component. It is a specification of behavior between outputs and inputs. Includes calculations, data manipulation, business logic. Example: "System shall charge the credit card on order confirmation."
Multi-SelectQ7 / 30A, C, D
7. Non-Functional Requirements (NFRs) are also referred to as which of the following? Select all that apply.
Explanation: NFRs have 3 alternate names: Architecturally Significant Requirements + System Quality Attributes + "ilities" (scalability, availability, maintainability, etc.). Categories include: Security, Scalability, Availability, Usability, Testability, Maintainability.
T/FQ8 / 30True
8. DevOps is the practice of operations and development engineers participating together in the entire service lifecycle, from design through development to production support.
Explanation: TRUE. This is the practical definition of DevOps. Key: it's about developers AND operations working TOGETHER across the ENTIRE lifecycle — not just at deployment time.
Multi-SelectQ9 / 30B, C, E
9. Identify the THREE primary practice areas of DevOps.
Explanation: 3 DevOps practice areas: (1) Infrastructure Automation — infra as code in SCM. (2) Site Reliability Engineering — monitoring, orchestration, design for operability. (3) Continuous Delivery — automating build, test, deploy.
MCQ10 / 30B
10. A continuous integration server is where (select the best answer):
Explanation: CI = triggered on every code change (every commit/push to the repo). NOT on a fixed schedule. This gives developers immediate feedback when they break something.
Multi-SelectQ11 / 30B, C, D, E
11. What should go in your project's single source code repository? Select all that apply.
Explanation: SCM repo = Code + Test Scripts + DB Schema + Install Scripts. NOT compiled binaries (use an artifact repo like Nexus) and NOT the OS (too large, managed separately).
MCQ12 / 30B
12. What is the Jenkins web server / "brains" of the system called?
Explanation: Controller = Jenkins web server + brains. Agent ≈ Node = manages task execution on a machine. Executor = runs stages in a pipeline. All three terms appear in Jenkins architecture questions.
MCQ13 / 30C
13. What runs the stages in a Jenkins pipeline?
Explanation: Executor = runs pipeline stages. Nick got this wrong on quiz 4 (selected Agent instead of Executor). Agent manages task execution on a node, but the Executor is the thread that actually runs the stages.
MCQ14 / 30D
14. In Jenkins declarative pipeline syntax, the pipeline is organized into:
Explanation: Declarative pipeline: pipeline { agent any; stages { stage('X') { steps { ... } } } }. "node block" = scripted pipeline syntax. Declarative uses "pipeline block".
T/FQ15 / 30False
15. It is generally considered best practice to define and maintain the Jenkins pipeline definition directly within a Jenkins build job.
Explanation: FALSE. Best practice = define the pipeline in a Jenkinsfile and check it into source control. Benefits: single source of truth, audit trail, code review, automatically creates pipeline for all branches/PRs.
MCQ16 / 30C
16. Shared libraries for Jenkins pipelines are written in which language?
Explanation: Jenkins Shared Libraries = Groovy. Functions live in /vars/*.groovy. Import: @Library('my-lib') _. Then call: myFunction(arg).
T/FQ17 / 30False
17. Global Pipeline Libraries in Jenkins are only available to specific pipeline jobs in the Jenkins server.
Explanation: FALSE. Global Pipeline Libraries = available to ALL pipeline jobs on the Jenkins server. That's what makes them "global". Non-global libraries can be scoped to specific jobs.
MCQ18 / 30B
18. Static code analysis is best described as:
Explanation: Static = WITHOUT execution. Dynamic = during/after execution. "Automated inspection" is not wrong but is not the defining characteristic — the key distinction is "without running the code".
MCQ19 / 30C
19. Static and dynamic code analysis together are referred to as:
Explanation: Glass box = Static + Dynamic. White box = testing with knowledge of internal program flow/logic. Black box = testing interfaces without knowing internals. Nick got this wrong on quiz 6 — common trap!
Multi-SelectQ20 / 30A, C, E
20. What are the types of issues that SonarQube can track in your code? Select all that apply.
Explanation: SonarQube tracks exactly 3 issue types: Vulnerabilities (security risks) + Bugs (defects likely to cause runtime errors) + Code Smells (maintainability issues). NOT "Errors" or "Warnings".
T/FQ21 / 30False
21. A SonarQube Quality Gate should initially be set so that ALL code — new and old — must meet the standard.
Explanation: FALSE. Quality Gates should initially apply to NEW code only. Old/legacy code may have technical debt that's impractical to fix immediately. Start with new code standards, then gradually address old code.
MCQ22 / 30B
22. In SonarQube, what best describes a scanner?
Explanation: Scanner = client app that runs against your code and computes snapshots. Server = stores config and results. SonarLint = IDE plugin for early feedback. First analysis should happen in the IDE (SonarLint), not the server.
T/FQ23 / 30True
23. The .gitlab-ci.yml file in GitLab CI is analogous to the Jenkinsfile in Jenkins.
Explanation: TRUE. Both files define CI/CD pipelines in source control. Stages in .gitlab-ci.yml = stages in Jenkinsfile. Stages can have jobs (GitLab) / steps (Jenkins) that run in parallel.
T/FQ24 / 30False
24. In a pipeline defined in a .gitlab-ci.yml file, each stage belongs to a job.
Explanation: FALSE. Each JOB belongs to a STAGE (not the reverse). A stage can contain multiple jobs. Jobs within the same stage run in parallel by default. Nick got this wrong on quiz 7.
MCQ25 / 30C
25. What are the agents that run your jobs in GitLab CI called?
Explanation: Runners = GitLab's equivalent of Jenkins Agent Nodes. Can be self-managed (your own infra) or GitLab-hosted. Shell executor = machine dependencies. Docker executor = containerized jobs.
MCQ26 / 30C
26. What advantage does an artifact repository have over storing artifacts as plain files on a shared filesystem?
Explanation: The key advantage = versioning. Artifact repositories (Nexus, Artifactory) version every build artifact so you can retrieve any specific version. Also proxy public repos (Maven Central, Docker Hub) to insulate from outages and speed up builds.
Multi-SelectQ27 / 30A, C, D
27. Identify THREE benefits of adopting Continuous Delivery practices.
Explanation: CD benefits: Higher quality (build quality in from the start) + Higher revenues (faster delivery of value) + Low risk releases (small frequent deployments + blue/green rollback). NOT "lower costs" or "fewer developers" — those weren't the stated benefits.
MCQ28 / 30B
28. In Trunk-Based Development, developers branch:
Explanation: Trunk-based = branch only when absolutely necessary. All changes return to mainline quickly. Benefits: eliminates divergence, minimizes merge conflicts, developers move fast. Best for experienced teams and fast-paced new product development.
T/FQ29 / 30False
29. A "Pull Request" is the GitLab term for requesting to merge one branch into another.
Explanation: FALSE. Pull Request = GitHub terminology. GitLab uses "Merge Request (MR)". Same concept: propose changes in one branch to be merged into another, with code review and CI pipeline before merging.
MCQ30 / 30C
30. Which best describes appropriate use cases for Jenkins vs GitLab CI?
Explanation: Jenkins = larger organizations with diverse/complex build requirements and configurations. GitLab CI = easier adoption for smaller teams already using GitLab and preferring a SaaS solution. This was quiz 11's final question (100% on that quiz).
