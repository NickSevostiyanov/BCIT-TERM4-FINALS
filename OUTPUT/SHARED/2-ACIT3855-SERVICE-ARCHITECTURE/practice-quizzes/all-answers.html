Part 1 — Microservices Fundamentals (Q1–Q4)
Q1 — Benefits of MicroservicesSelect All That Apply
Identify the BENEFITS of a Microservices Architecture (select all that apply).
✓Technology Diversity — mix languages, frameworks, databases per service
○Distribution — this is a COST
✓Strong Module Boundaries — enforces modular structure for large teams
○Eventual Consistency — this is a COST
✓Independent Deployment — deploy one service without touching others
Benefits: Technology Diversity, Strong Module Boundaries, Independent Deployment. Distribution (remote calls are slow/fallible) and Eventual Consistency (can't use DB transactions across services) are COSTS. Quiz 1 directly tested this distinction — memorize it.
Q2 — Costs of MicroservicesSelect All That Apply
Identify the COSTS of a Microservices Architecture.
○Independent Deployment — BENEFIT
✓Eventual Consistency — no cross-service DB transactions
○Technology Diversity — BENEFIT
✓Operational Complexity — need mature ops team
✓Distribution — remote calls are slow and can fail
Costs: Eventual Consistency, Operational Complexity, Distribution. Independent Deployment is a benefit (not a cost). Technology Diversity is a benefit. From Martin Fowler's Microservices guide — tested in Quiz 1.
Q3 — Service APISingle Best Answer
What is the API of a service typically implemented as?
○Class Methods
○Instance Methods
○Public Methods
✓HTTP Methods
HTTP Methods. Services expose their APIs via HTTP (GET/POST/PUT/DELETE). RESTful. Other method types are in-process and not service boundaries.
Q4 — Business CapabilitiesTrue/False
Microservices are typically organized around business capabilities.
✓True
○False
TRUE. Microservices = organized by business capability (Order Service, Payment Service). Monoliths = often organized by tech layer (UI team, DB team). Conway's Law: org structure mirrors system design.
Part 2 — Connexion & OpenAPI (Q5–Q8)
Q5 — operationIdSingle Best Answer
How do you route an OpenAPI endpoint to a Python function in Connexion?
○path
○function
✓operationId
○paths
operationId. In the YAML: operationId: app.my_function → calls my_function() in app.py. Without this, Connexion won't know which Python function to call for that endpoint. Tested in Quiz 1.
Q6 — strict_validationSingle Best Answer
What parameter enables automatic request parameter validation in Connexion?
○validator_map
○validation
✓strict_validation=True
○validate_responses (different flag — for response validation)
strict_validation=True. Pass this to app.add_api('openapi.yaml', strict_validation=True). Automatically validates request parameters. Tested in Quiz 1 — the trap is "validation" (wrong) vs "strict_validation" (correct).
Q7 — ComponentsSingle Best Answer
Why use components in an OpenAPI 3.0 specification?
○Organization
✓Reuse
○Supports multiple properties
○It's required
Reuse. Define a schema once under components/schemas, reference it anywhere with $ref: '#/components/schemas/MySchema'. Components are optional, not required. Tested in Quiz 4.
Q8 — YAML vs JSONTrue/False
OpenAPI specs should be written in JSON — it's easier than YAML.
○True
✓False
FALSE. YAML is recommended because it's more human-readable (less punctuation). Both are supported. The course uses YAML for all OpenAPI specs. Tested in Quiz 4.
Part 3 — Data & Persistence (Q9–Q10)
Q9 — Polyglot PersistenceSingle Best Answer
Best definition of Polyglot Persistence in Microservices?
✓Each Service Manages Its Own Database
○Each Service Uses a Different Programming Language
○Each Service Uses a NoSQL DB
○Each Service Must Use A Different Database Technology
Each service manages its own database. Private, accessible only via its API. Services CAN use different technologies but don't HAVE to. "Different language" = polyglot (not polyglot persistence). Services can use the SAME DB technology if it meets their needs.
Q10 — Monolith & PolyglotTrue/False
A Monolith cannot use Polyglot Persistence.
○True
✓False
FALSE. A monolith CAN use polyglot persistence (e.g., different tables using different storage engines) — just less common. Polyglot persistence is more naturally enforced in microservices due to service isolation. Tested in Quiz 2.
Part 4 — Logging, Tracing & Config (Q11–Q13)
Q11 — Event LoggingSingle Best Answer
Characteristics of Event Logging?
○Consumed by developers (that's tracing)
○Low-level info, can be noisy (that's tracing)
✓High-level info, system administrators, must not be noisy
○Used for debugging (that's tracing)
Event Logging: high-level, sys admins, not noisy. Software Tracing: low-level, developers, CAN be noisy, for debugging. Two completely different purposes. Tested in Quiz 3.
Q12 — Config File FormatsSelect All That Apply
Valid configuration file formats for a Python program:
✓XML
✓INI
✓YAML
✓Python
✓JSON
○CSV — for data, not config
All 5 (XML, INI, YAML, Python, JSON). This was tested exactly in Quiz 3 — "select all that apply". CSV is for tabular data, not configuration. YAML is the most common in this course (app_conf.yml, log_conf.yml).
Q13 — Logging LevelSingle Best Answer
What does the level mean in the Python logging config?
✓The minimum severity logging level reported
○The maximum severity before crash
○The number of logging levels
○The maximum severity reported
Minimum severity (floor). level=WARNING → only WARNING, ERROR, CRITICAL logged. DEBUG and INFO ignored. Order: DEBUG(10) < INFO(20) < WARNING(30) < ERROR(40) < CRITICAL(50). Tested in Quiz 3.
Part 5 — Messaging & Kafka (Q14–Q17)
Q14 — REST is SynchronousSingle Best Answer
RESTful APIs using HTTP is a form of:
○Asynchronous Communication
✓Synchronous Communication
○Pub/Sub Messaging
○Decoupling
Synchronous. REST: caller sends request, WAITS for response. Blocking. Messaging/Kafka: fire-and-forget, asynchronous. This distinction is fundamental to the course. Tested in Quiz 5.
Q15 — Messaging is AsynchronousSingle Best Answer
Messaging (e.g., Kafka) is a form of:
○Synchronous Communication
○Database Communication
✓Asynchronous Communication
○Coupling
Asynchronous. Producer sends to broker and moves on. Consumer reads independently. Decoupled in time. Tested in Quiz 5.
Q16 — Messaging AdvantagesSelect All That Apply
Advantages of using messaging when deploying microservices (select 4):
✓Scalability — add consumers to handle more load
✓High-Availability — messages persist even if consumer is down
✓Producer/Consumer Decoupling — producers don't know about consumers
○Producer/Consumer Coupling — opposite of what messaging provides
✓Publish/Subscribe Pattern — broadcast to multiple consumers
○High-Security — not a messaging advantage
4 correct: Scalability, High-Availability, Decoupling, Pub/Sub. Coupling is the PROBLEM messaging solves (not a benefit). High-Security is not a feature of messaging systems. Tested in Quiz 5.
Q17 — Kafka DeploymentTrue/False
Kafka can only be deployed on a single stand-alone machine.
○True
✓False
FALSE. Kafka can run as a CLUSTER of brokers for high-availability. Each broker handles a subset of topic partitions. If one broker fails, others continue serving. Single-broker is only for dev/testing.
Part 6 — Docker (Q18–Q20)
Q18 — Volume LifecycleTrue/False
A Docker volume is automatically removed when the container is removed.
○True
✓False
FALSE. Volumes persist after container removal — that's the point. You need docker volume rm <name> or docker-compose down -v to remove them. Volumes auto-CREATE when container starts (if they don't exist), but do NOT auto-delete. Tested in Quiz 7.
Q19 — Mount TypesSelect All That Apply
Three types of mounts for Docker containers:
✓tmpfs — stores in host memory only (not persisted)
○source — not a mount type
✓volume — managed by Docker, persists independently
✓bind — mounts file/directory from host VM
○volume-opt — not a mount type
volume, bind, tmpfs. Volume: Docker-managed, best for prod data. Bind: maps host path into container (useful for dev, config files). tmpfs: in-memory only, not persisted (useful for secrets/temp data). Tested in Quiz 7.
Q20 — Container FilesystemSingle Best Answer
Container filesystem data is deleted when:
○When stopped
○When image is built
✓When the container is REMOVED
○When image is removed
Removed (not stopped). Stopped container = filesystem intact, can restart. Removed container = filesystem gone. This is why volumes exist — to persist data outside the container lifecycle. Tested in Quiz 7.
Part 7 — Deployment Strategies (Q21–Q26)
Q21 — Blue/Green ColorsTrue/False
In blue/green deployment: Blue = NEW version, Green = EXISTING version.
✓True
○False
TRUE — Blue = NEW. The Quiz 10 question stated the OPPOSITE as a trap (said blue=new was false). The correct answer is: Blue IS the new version, Green IS the existing. Easy rollback: route traffic back to green if blue has issues.
Q22 — NGINX DefaultSingle Best Answer
Default load balancing strategy for NGINX?
○IP Hash
○Least Connected
✓Round Robin
○Random with Two Choices
Round Robin. Distributes requests sequentially: server1, server2, server3, server1... Simple, effective for evenly loaded servers. Must be memorized — tested in Quiz 10.
Q23 — Canary TestersSingle Best Answer
Who tests a canary deployment in production?
○Developer
○Sales Team
○Testers
✓End Users
End Users. Canary = small % of real production traffic routes to new version. Real users unknowingly test it. Monitor for errors — if OK, gradually increase to 100%. This is NOT a staging environment. Tested in Quiz 10.
Q24 — Docker Compose StrategySingle Best Answer
Which deployment strategy does Docker Compose provide?
○Ramped
✓Recreate
○Blue/Green
○Canary
Recreate. docker-compose up stops all containers then starts new ones. Service is DOWN during update. For zero-downtime deployments (blue/green, canary, ramped), you need Kubernetes or more advanced tooling. Tested in Quiz 10.
Q25 — Recreate HA?True/False
The Recreate strategy for Kubernetes is high-availability.
○True
✓False
FALSE. Recreate stops ALL existing instances BEFORE starting new ones → downtime. Not high-availability. HA strategies keep at least some instances running during the update. Tested in Quiz 10.
Q26 — NGINX Hardware?True/False
NGINX is an example of a hardware load balancer.
○True
✓False
FALSE — NGINX is software. Hardware load balancers are dedicated physical appliances (F5, Citrix). Software LBs (NGINX, HAProxy, Envoy) run on standard servers. This was tested in Quiz 10.
Part 8 — Testing, Tracing & Technical Debt (Q27–Q30)
Q27 — Testing TypesSelect All That Apply
Automated testing segments in microservices (select 4):
○Database Validity Testing — not in reading
✓End to End Testing
✓UI Functional Testing
✓Unit Testing
✓Integration Testing
○Code Branch Testing — not standard term from reading
4 correct: End to End, UI Functional, Unit, Integration. Standard testing pyramid for microservices. Unit = smallest; Integration = components together; UI Functional = browser behavior; E2E = full system. Tested in Quiz 9.
Q28 — Distributed TracingTrue/False
Observations of a single event: Receiver → Storage → Processing = Distributed Tracing.
✓True
○False
TRUE. Distributed Tracing = tracking ONE request/event as it crosses multiple service boundaries. Advantages: end-to-end visibility, faster developer debugging. Tested in Quiz 9.
Q29 — Round Robin AlgorithmSingle Best Answer
What algorithm best describes Round Robin load balancing?
○Picks two randomly, uses least connections — that's "Random with Two Choices"
○Fewest current connections — that's "Least Connected"
✓Requests distributed across servers sequentially
○Client IP determines server — that's "IP Hash"
Sequential distribution. 1→A, 2→B, 3→C, 4→A again. Simple, balanced under even load. Know ALL 4 algorithms: Round Robin (sequential), Least Connected (fewest connections), IP Hash (client IP), Random with Two Choices. Tested in Quiz 10.
Q30 — Kafka Connection LossSingle Best Answer
When a Producer/Consumer loses connection to Kafka, best practice is:
○Log error and do nothing
✓Restart the consumer/producer object to re-establish connection
○Restart the entire Docker container
○Switch to REST API
Restart the producer/consumer object. Catch the exception, then recreate the Producer or Consumer. Producer exceptions: LeaderNotAvailable, SocketDisconnectedError. Consumer exception: SocketDisconnectedError. Restarting the whole container is overkill and causes downtime. Tested in Quiz 9.
