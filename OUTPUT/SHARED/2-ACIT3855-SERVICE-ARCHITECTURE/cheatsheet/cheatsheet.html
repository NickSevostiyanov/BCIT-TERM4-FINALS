1. MICROSERVICES vs MONOLITH (Martin Fowler)

Benefits of Microservices ✓

  • Technology Diversity — mix languages, frameworks, DBs per service (polyglot)
  • Strong Module Boundaries — enforces modularity, important for large teams
  • Independent Deployment — deploy one service without touching others
  • Scalability — scale ONLY the bottleneck service

Costs of Microservices ✗

  • Eventual Consistency — no cross-service DB transactions; everyone manages it
  • Operational Complexity — need mature ops team, many services being redeployed
  • Distribution — remote calls slow, always at risk of failure

Key Terms

  • Polyglot — each service uses different language/tech
  • Conway's Law — system design mirrors org communication structure
  • Tech-layer orgs → time-consuming approval process
  • Business capability teams → better autonomy

True/False Traps

2. CONNEXION + OPENAPI 3.0

app.py — Minimal Connexion Setup

import connexion

app = connexion.FlaskApp(__name__, specification_dir='')
app.add_api(
    'openapi.yaml',
    strict_validation=True,   # validates REQUEST params ← KEY
    validate_responses=True   # validates RESPONSES (separate flag)
)

def init_scheduler():
    from apscheduler.schedulers.background import BackgroundScheduler
    sched = BackgroundScheduler(daemon=True)
    sched.add_job(populate_stats, 'interval',
                  seconds=app_config['scheduler']['period_sec'])
    sched.start()

if __name__ == "__main__":
    init_scheduler()
    app.run(port=8080, use_reloader=False)

Connexion Key Facts

  • operationId: app.my_function → routes endpoint to Python function
  • strict_validation=True → validates REQUEST parameters ← most tested
  • validate_responses=True → validates RESPONSES (different flag!)
  • apscheduler runs in background daemon thread
  • Period stored in app_conf.yml (not hardcoded)
  • Both API + scheduler run in same application
  • Use use_reloader=False to prevent double scheduler start
⚠ Trap: validation="validation" is WRONG. Must be strict_validation=True

            format: date-time

            type: string
            format: date-time

          content:

3. DATA PERSISTENCE

Polyglot Persistence

  • Each service manages its OWN DB
  • Data accessible ONLY via service API
  • Transactions only involve THEIR DB (not central)
  • Services CAN use different DB tech (but don't HAVE to)
  • Monolith CAN use polyglot persistence (less common)
⚠ "Each service must use different DB tech" = FALSE. They CAN, not must.

DB-Per-Service Pattern

Valid SQL options:

  • Private-tables-per-service
  • Schema-per-service
  • Database-server-per-service
  • ❌ Datafile-per-service — NOT valid SQL option

Benefits: loosely coupled, use best-fit DB

Drawbacks: complex multi-DB mgmt, cross-service transactions hard, cross-DB queries challenging

5. MESSAGING & APACHE KAFKA

REST/HTTP vs Messaging — KEY DISTINCTION

REST/HTTPMessaging (Kafka)
TypeSynchronousAsynchronous
PatternRequest/ReplyFire-and-forget
WaitingCaller waitsCaller moves on
CouplingTighterLoose (decoupled)
Good forSOA, public APIs, request/replyEvents, multi-consumer, async

Why REST doesn't scale for microservices:

  • Cannot deliver event to multiple places simultaneously
  • Cannot deliver messages asynchronously
  • Doesn't scale as more applications/instances added

Messaging Advantages (4)

  • Scalability — add consumers to handle more load
  • High-Availability — messages persist if consumer is down
  • Producer/Consumer Decoupling — producers don't know consumers
  • Publish/Subscribe Pattern — broadcast to multiple consumers

NOT advantages: Producer/Consumer Coupling (opposite!), High-Security

Event-Driven Architecture

Producers emit business events → messaging system → consumers subscribe to events they want

Helps decouple services and enables scalability

Kafka Concepts

  • Topic — stream of records (key-value); each assigned an offset (sequence number)
  • Producer — publishes messages to topics
  • Consumer — subscribes to topics, reads messages
  • Broker — instance of Kafka responsible for message exchange
  • Cluster — multiple brokers for HA (NOT single machine only)

Kafka Features & Use Cases

Features (4): Low-Latency, High-Throughput, Distributed, Massively Scalable

Use Cases (4): Activity Monitoring, Log Aggregation, Messaging, Database

NOT use cases: Application Config Store, Data Visualization

Kafka Error Handling

Producer connection loss → catch:

  • LeaderNotAvailable
  • SocketDisconnectedError

Consumer connection loss → catch:

  • SocketDisconnectedError

Best practice: Restart the producer/consumer OBJECT (not whole container)

Kafka Python Code

# Producer
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers='kafka:9092'
)

msg = {
    "sensor_id": "abc123",
    "value": 42.5,
    "timestamp": "2026-01-15T10:00:00Z"
}

try:
    producer.send(
        'events',
        json.dumps(msg).encode('utf-8')
    )
    producer.flush()
except Exception as e:
    logger.error(f"Failed to send: {e}")
# Consumer
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'events',
    bootstrap_servers='kafka:9092',
    auto_offset_reset='earliest',
    group_id='processing-service'
)

for msg in consumer:
    data = json.loads(msg.value.decode('utf-8'))
    logger.info(f"Received: {data}")
    process(data)

7. SAME ORIGIN POLICY (SOP) & CORS

Same Origin Policy (SOP)

  • Browser security feature — prevents cross-origin requests by default
  • Origin = scheme + host + port (all 3 must match)
  • Example: http://app.com:3000 ≠ http://api.com:8080 (different host + port)
  • Only applies to browser-initiated requests (not server-to-server)
  • Problem: frontend on port 3000 cannot call backend on port 8080 without CORS

CORS (Cross-Origin Resource Sharing)

  • Mechanism to relax SOP when cross-origin requests are needed
  • Server adds response headers to allow specific origins/methods

Key Headers:

  • Access-Control-Allow-Origin: * or specific origin
  • Access-Control-Allow-Methods: GET, POST, PUT, DELETE
  • Access-Control-Allow-Headers: Content-Type, Authorization

Preflight request: Browser sends OPTIONS request first for non-simple requests. Server must respond with CORS headers for browser to proceed.

# Flask-CORS setup
from flask_cors import CORS
CORS(app)
# or specific origins:
CORS(app, origins=['http://localhost:3000'])

8. DEPLOYMENT STRATEGIES

Recreate ❌ HA

  • Stop ALL old instances
  • THEN start new ones
  • Service DOWN during update
  • Docker Compose uses this
  • Kubernetes: NOT HA

Ramped (Rolling) ✓ HA

  • Gradually replace old with new
  • 1-by-1 or in batches
  • Always some instances running
  • Slow rollout
  • Kubernetes: default

Blue/Green ✓ HA

  • Blue = NEW version
  • Green = EXISTING version
  • Both run in parallel
  • Switch traffic from green→blue
  • Easy rollback to green
  • High-availability ✓

Canary ✓ HA

  • Small % traffic → new version
  • Tested by End Users
  • Monitor for errors
  • Gradually increase to 100%
  • Real production testing
⚠ TRAPS: "Blue/Green: Blue=old, Green=new" = FALSE (it's reversed). "Docker Compose provides blue/green" = FALSE (provides Recreate). "Recreate = HA" = FALSE.

9. LOAD BALANCING (NGINX)

NGINX Facts

  • NGINX = SOFTWARE load balancer (NOT hardware)
  • Hardware LBs are dedicated physical appliances (F5, Citrix)
  • Default strategy: Round Robin
  • docker-compose up --scale service=3 → 3 replicas

Non-functional characteristics a LB provides:

  • Availability — redirect if server down
  • Scalability — add/remove servers from pool

3 Primary LB functions:

  • Distribute traffic across servers
  • Send only to online servers
  • Allow adding/removing servers based on load

Load Balancing Algorithms

AlgorithmHow it works
Round Robin (DEFAULT)Requests distributed sequentially: 1→A, 2→B, 3→C, 4→A...
Least ConnectedSend to server with fewest active connections
IP HashClient IP determines server (session persistence)
Random w/ Two ChoicesPick 2 at random, use Least Connections of those 2
HashUser-defined key determines server
Least TimeFastest + fewest connections
⚠ MUST memorize: NGINX default = Round Robin. "Requests distributed sequentially" = Round Robin definition.

    session.commit()
    session.close()

def get_readings(start_timestamp, end_timestamp):
    """ GET /readings """
    session = DB_SESSION()
    results = session.query(Reading).filter(
        Reading.timestamp >= start_timestamp,
        Reading.timestamp <= end_timestamp
    )
    results_list = []
    for r in results:
        results_list.append(r.to_dict())
    session.close()

    logger.info(f"Returning {len(results_list)} readings")
    return results_list, 200
